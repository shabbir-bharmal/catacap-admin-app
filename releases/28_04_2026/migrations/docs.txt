Migration: 2026_04_28_daf_providers.sql
Release date: 2026-04-28
Feature: DAF Providers master table + seed (Pending Grants filter fix)

Overview
--------
The Node.js Postgres-backed admin Site Configuration and Pending
Grants routes already read from / write to a `daf_providers` table:

  - `server/src/routes/siteConfiguration.ts` (`SELECT id,
    provider_name, provider_url, is_active FROM daf_providers
    ORDER BY provider_name`, plus `INSERT` / `UPDATE` /
    toggle-active by id).
  - `server/src/routes/adminPendingGrants.ts` (`SELECT
    LOWER(TRIM(provider_name)) FROM daf_providers` to compute the
    "Other" branch of the DAF provider filter).

That table never existed in Postgres — the canonical seed lived
only in the .NET backend at
`Back-End/Invest.Repo/Data/DAFProviderData.cs`. As a result:

  - `GET /api/admin/site-configuration/daf-providers` returned
    HTTP 500.
  - The Pending Grants admin page DAF Provider filter dropdown
    showed only "All Providers" (and the synthetic "Other" entry
    the frontend appends), no real providers.
  - The Site Configuration → DAF Providers tab showed an empty
    list with no way to add the canonical providers.

This migration creates the table and seeds it from the .NET source
of truth so the existing route code starts working without any
backend route changes.

Schema
------
CREATE TABLE daf_providers (
    id            SERIAL PRIMARY KEY,
    provider_name TEXT    NOT NULL,
    provider_url  TEXT,
    is_active     BOOLEAN NOT NULL DEFAULT TRUE
);

Columns and types are chosen to match exactly what the existing
Node.js routes already reference (no rename / cast needed in the
route layer).

Seed data
---------
Mirrors `Back-End/Invest.Repo/Data/DAFProviderData.cs`, including
the integer `id` value for each row so the two backends stay
aligned:

  1  Fidelity Charitable
  2  Jewish Foundation
  3  ImpactAssets
  4  National Philanthropic Trust
  5  DAFgiving360: Charles Schwab
  6  Silicon Valley Community Foundation
  7  Vanguard Charitable
  8  Bay Area Jewish Federation

All rows seeded with `is_active = TRUE`.

Idempotency
-----------
- `CREATE TABLE IF NOT EXISTS` so the table-create step is safe to
  re-run.
- The seed `INSERT` is gated by a `NOT EXISTS` check that compares
  `LOWER(TRIM(provider_name))` against the incoming row, so:
    * Re-running won't duplicate any of the 8 seeded rows.
    * If an admin has already created one of the seeded providers
      manually through the Site Configuration UI under a slightly
      different casing or with surrounding whitespace, the seed
      will treat it as already present and leave it alone.
- After seeding, `daf_providers_id_seq` is reset (via `setval`
  with `is_called = TRUE`) to `MAX(id)`, so the next `INSERT` from
  the Site Configuration "Add DAF Provider" form gets a fresh id
  that will not collide with the seeded ids 1–8.
- Wrapped in a single transaction so a partial failure rolls back
  cleanly.

What it deliberately does NOT do
--------------------------------
- It does NOT backfill or normalize the `daf_provider` text column
  on `pending_grants`. Existing grant rows keep whatever provider
  name string they were written with; the filter logic in
  `adminPendingGrants.ts` already does case-insensitive / trimmed
  matching against this seed list.
- It does NOT change the .NET backend or its `DAFProviders` seed
  data — those remain the source of truth for the .NET side.
- It does NOT add a foreign key from `pending_grants.daf_provider`
  (a free-text column) to `daf_providers.provider_name`.

Application-side behavior (not in this migration)
-------------------------------------------------
- The Pending Grants frontend (`src/pages/PendingGrants.tsx`) no
  longer appends a synthetic "Other" entry to the DAF Provider
  dropdown. The dropdown now shows just "All Providers" plus the
  rows returned by the API, sorted alphabetically.
- A new "Foundation Grants" checkbox on the Pending Grants filter
  row, when checked, disables the DAF Provider dropdown and tells
  the API to return only foundation grants. The frontend sends
  `foundationGrantsOnly=true`; the Node.js
  `GET /api/admin/pending-grant` route adds
  `LOWER(TRIM(pg.daf_provider)) = 'foundation grant'` to the
  WHERE clause and ignores any incoming `dafProvider` value.
- The "Other" branch of `adminPendingGrants.ts` is intentionally
  left in place. It is no longer reachable from the Pending Grants
  dropdown after this change (the synthetic "Other" entry is gone)
  but is preserved so other / future callers are unaffected.

Rollback
--------
A commented rollback block at the bottom of the SQL file shows how
to drop `daf_providers`. Apply the rollback BEFORE rolling the
application code back to a build that does not reference the
table — though note that the Node.js routes were already
referencing the table before this migration (and silently failing
with 500), so a rollback simply returns the system to that broken
state.

Migration: 2026_04_28_event_show_on_home.sql
Release date: 2026-04-28
Feature: Event form: "Show on Home Page" toggle

Overview
--------
Adds a new `show_on_home` boolean column to the `events` table so admins
can opt out of having a given event appear on the public CataCap home
page without taking the event down (the existing `status` Active/Draft
toggle is unchanged and remains the on/off switch for the admin
listing).

The new column is paired with:

  - A new "Show on Home Page" Yes/No radio group on the admin Event
    Management create/edit dialog
    (`src/pages/EventManagement.tsx`), defaulting to Yes for new
    events and reflecting the saved value when editing.
  - A new optional `showOnHome: boolean` field on the
    `EventApiItem` and `EventCreateUpdatePayload` types in
    `src/api/event/eventApi.ts`.
  - Read / write support in the Node.js admin route
    (`server/src/routes/events.ts`) for list, get-by-id, insert,
    and update.
  - A new filter (`AND show_on_home = true`) on the public
    upcoming events route (`server/src/routes/publicEvents.ts`).
    The admin Event Management list intentionally does NOT apply
    this filter, so admins can still see and edit events with
    "Show on Home Page = No".

Schema
------
ALTER TABLE events
    ADD COLUMN IF NOT EXISTS show_on_home BOOLEAN NOT NULL DEFAULT TRUE;

Defaulting to TRUE matters for two reasons:
  1. Existing rows are backfilled to TRUE on add, so nothing
     disappears from the public home page after the migration runs.
  2. New rows inserted by application code that does not specify
     `show_on_home` (e.g. a future caller, or a partial payload)
     continue to appear on the home page by default, matching the
     admin form's default selection.

Idempotency
-----------
- `ADD COLUMN IF NOT EXISTS` so re-running the migration is a no-op.
- Wrapped in a single transaction so a partial failure rolls back
  cleanly.

Application-side behavior (not in this migration)
-------------------------------------------------
- The admin Event Management form
  (`src/pages/EventManagement.tsx`) also gains a Duration format
  validator that accepts patterns like `30m`, `1h`, `1h 30m`, and
  `2h 15m` (case-insensitive, whitespace tolerated, empty still
  allowed) and shows an inline error / blocks save on invalid
  entry. That UI change is independent of this migration but is
  part of the same task / release.
- The .NET backend under `Back-End/` is reference-only and is NOT
  updated in this release; the running app uses the Node.js +
  Postgres stack.

Rollback
--------
A commented rollback block at the bottom of the SQL file shows how
to drop the column. Apply the rollback BEFORE rolling the
application code back to a build that does not reference the
column.

Migration: 2026_04_28_event_show_on_home_nullable.sql
Release date: 2026-04-28
Feature: Event form: "Show on Home Page" toggle becomes tri-state

Overview
--------
Per product decision after the initial rollout, the
`events.show_on_home` column must be tri-state:

  - TRUE  — admin explicitly chose Yes (event appears on the
            public home page).
  - FALSE — admin explicitly chose No (event hidden from the
            public home page but still visible in the admin list
            and reachable via its registration link).
  - NULL  — admin has not made an explicit choice yet
            (e.g. legacy rows imported from another system, or
            rows created by a non-form caller that did not set
            the field). Treated as "do not show on home" by the
            public endpoint, same as FALSE.

The earlier same-day migration
(`2026_04_28_event_show_on_home.sql`) added the column as
`BOOLEAN NOT NULL DEFAULT TRUE`, which forced every row to be
either TRUE or FALSE and silently coerced unspecified payloads
to TRUE.

Schema
------
ALTER TABLE events ALTER COLUMN show_on_home DROP NOT NULL;
ALTER TABLE events ALTER COLUMN show_on_home DROP DEFAULT;

Existing TRUE / FALSE values are preserved unchanged. New rows
inserted without an explicit `show_on_home` value are now
stored as NULL rather than silently defaulting to TRUE.

Idempotency
-----------
- `ALTER COLUMN ... DROP NOT NULL` and `ALTER COLUMN ... DROP
  DEFAULT` are no-ops if the constraint / default has already
  been removed, so the migration is safe to re-run.
- Wrapped in a single transaction so a partial failure rolls
  back cleanly.

Application-side behavior (not in this migration)
-------------------------------------------------
- `server/src/routes/events.ts` save handler now writes
  `dto.showOnHome === true ? true : dto.showOnHome === false ?
  false : null` for both INSERT and UPDATE, so a payload that
  omits the field stores NULL instead of being silently coerced
  to FALSE (or to the previous TRUE default). The admin Event
  Management form already requires the admin to pick Yes or No
  before save, so the legitimate save path always sends a real
  boolean; the NULL branch is the safety net for non-form
  callers.
- `server/src/routes/events.ts` list and get-by-id responses
  now emit `showOnHome: r.show_on_home ?? null` (instead of
  `?? false`), so the admin form opens with no Yes/No selected
  for legacy / NULL rows and the admin must make an explicit
  choice on the next edit.
- `server/src/routes/publicEvents.ts` continues to filter on
  `show_on_home = true` in the WHERE clause, so NULL rows are
  excluded from the public home page (same end-user behavior
  as FALSE). The response payload also passes `showOnHome`
  through as `r.show_on_home ?? null`.
- `src/api/event/eventApi.ts` widens both `EventApiItem` and
  `EventCreateUpdatePayload` to `showOnHome?: boolean | null`
  so the tri-state value is type-correct end-to-end.
- The .NET backend under `Back-End/` is reference-only and is
  NOT updated in this release.

Rollback
--------
A commented rollback block at the bottom of the SQL file shows
how to restore the NOT NULL constraint and DEFAULT TRUE (after
backfilling any NULL rows to TRUE). Apply the rollback BEFORE
rolling the application code back to a build that does not
treat `show_on_home` as a tri-state value.

Migration: 2026_04_28_note_attachments.sql
Release date: 2026-04-28
Feature: Note attachment tables for Pending Grants and Other Assets

Overview
--------
The admin Pending Grants and admin Other Assets pages now support
file attachments on status-change notes. Each note can have zero or
more attached files, uploaded to Supabase Storage and recorded in
Postgres via two new tables:

  - `pending_grant_note_attachments`
    Attachments belonging to rows in `pending_grant_notes`.

  - `asset_based_payment_request_note_attachments`
    Attachments belonging to rows in `asset_based_payment_request_notes`.

The application also creates these tables on boot via
`ensureNoteAttachmentsTables()` in `server/src/db.ts`. This release
captures the same DDL as a tracked migration so the schema change is
visible in the release history (and so a fresh environment that does
not boot the Node service first still has the tables).

Schema
------
Both tables share the same shape:

  id            SERIAL PRIMARY KEY
  note_id       INTEGER NOT NULL  -- FK to the parent notes table
  file_name     TEXT NOT NULL     -- original uploaded filename
  storage_path  TEXT NOT NULL     -- path inside the storage bucket
  mime_type     TEXT
  size_bytes    BIGINT
  uploaded_at   TIMESTAMP DEFAULT NOW()
  uploaded_by   VARCHAR(450)      -- admin user id

  Index: (note_id) for fast lookups when rendering a note.

Foreign keys cascade-delete with the parent notes row, so removing
a note cleans up all of its attachment metadata.

Storage layout
--------------
Files are uploaded to the existing Supabase Storage bucket
(`SUPABASE_STORAGE_BUCKET`) under a new top-level `notes` folder:

  - notes/pending-grants/<uuid>.<ext>
  - notes/other-assets/<uuid>.<ext>

The `storage_path` column on each attachment row holds the bucket
key returned by the storage upload, so old/new path conventions
coexist without code changes.

Idempotency
-----------
- `CREATE TABLE IF NOT EXISTS` and `CREATE INDEX IF NOT EXISTS`
  make every step safe to re-run.
- Each table-create is wrapped in a `DO $$ ... $$` block that first
  checks for the parent notes table. If the parent table is not
  present in this environment, the corresponding attachments table
  is skipped — mirroring the runtime guard in
  `ensureNoteAttachmentsTables()` and matching the behaviour of
  environments that have only one of the two notes tables.
- Wrapped in a single transaction so a partial failure rolls back
  cleanly.

What it deliberately does NOT do
--------------------------------
- It does NOT create or modify the parent notes tables
  (`pending_grant_notes`, `asset_based_payment_request_notes`).
  Those are owned by their respective feature areas.
- It does NOT touch the storage bucket itself; the Supabase bucket
  and its policies are managed outside this migration.
- It does NOT delete any storage objects on rollback. If you drop
  the tables you may want to clean up `notes/pending-grants/` and
  `notes/other-assets/` in the bucket separately.

Rollback
--------
A commented rollback block at the bottom of the SQL file drops both
tables. Apply it BEFORE rolling the application code back to a
build that does not reference the tables.
